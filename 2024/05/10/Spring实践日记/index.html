IOC容器

IoC（Inversion of Control，控制反转）就是把原来代码里需要实现的对象创建、依赖，反转给容器来帮忙实现。我们需要创建一个容器，同时需要一种描述来让容器知道要创建的对象与对象的关系。这个描述最具体的表现就是我们所看到的配置文件。
org.springframework.beans和org.springframework.context包是 Spring Framework 的 IoC 容器的基础。该 BeanFactory接口提供了一种高级配置机制，能够管理任何类型的对象。 ApplicationContext 是BeanFactory的子接口。

IOC容器的两种实现方式（IDEA）

现在我们已经知道ApplicationContext是BeanFactory的子接口，并且也知道BeanFactory是用来管理容器的，由此得知ApplicationContext也是管理容器的，再推出ApplicationContext的实现类也是同样的作用。

  • IDEA向下查看实现类快捷键：ctrl+h（MAC上control+h）
  • IDEA向上查看接口和继承快捷键：CTRL+Alt+U

一般情况下，我们实现容器管理的方式大致分为XML和注解两种。XML是早年我们比较常用的方式，注解是我们现在流行使用的方式。

依赖注入

DI（Dependency Injection，依赖注入）就是指对象被动接受依赖类而不自己主动去找，换句话说，就是指对象不是从容器中查找它依赖的类，而是在容器实例化对象时主动将它依赖的类注入给它。

DI依赖注入主要方式：构造函数，setter，接口（废弃）

Bean的定义

IOC容器管理的就是Bean对象，但是这些对象也是各不相同，所以需要使用一些元数据定义，这样才能让我们区分对象的不同。举个生活中的例子：一般我们家里买汽车，虽然汽车都是一个车身+4个轮子组成，但是细分下来里面很多的配置都是不太一样的。品牌，引擎，型号等等。bean对象也是如此。

id属性

每个注册到IOC容器的对象都需要一个唯一标识来对其他对象进行区分，例如车编号或身份证ID一样。我们一般获取bean的方式都是通过id来进行获取。当然，除了id以外，也可以使用其他的方式获取Bean，例如name，alias（别名）。name可以使用id属性不能使用的一些字符，如斜杠（/），而且可以通过逗号，空格或冒号分割指定多个name，alias跟name使用方式基本一致。

注：若是指定@Bean注解，id为方法名，若是未指定，id默认为class类名，但首字母小写。

class属性

一般情况下，我们在注入的时候都需要指定Class，也就是我们当前的类（包名+类名）。

depends-on

一般情况下，我们对象之间都有依赖关系，比如我们在实例化对象A时，若对象A中有成员属性对象B，那么会先实例化对象B，之后再实例化对象A，这样最终实例化出来的对象A是一个完整对象。

Autowired

指定bean之间的依赖关系。Spring提供了4种自动注入的模式：no，byName，byType，constructor。一般来讲我们常用的就是@Autowired注解，Autowired注解是按照类型进行匹配。

当Spring根据注解Autowired在IOC容器中查找对应的对象实例，如果只有一个，那么就进行依赖注入。若存在多个的情况，如若不指定进一步匹配的方式，则会直接报错。

注：Autowired不但可以指定成员属性，还可以指定构造器，方法。

Qualiter

Qualiter注解是byName的方式，即当Autowired注解无法在多个实例对象中选取对应的实例，那么就通过Qualiter注解指定某个对象实例。

Primary

Primary指的是Bean注入的优先级，优先注入。同样也解决了IOC存在多个实例而无法抉择的问题。

Lazy(懒加载)

一般情况下，IOC容器会初始化所有的单例Bean，但是有时候我们并不是很迫切的需要这个bean，而且每次启动会把所有的单例Bean都加载，也会占据我们的内存资源和启动时间。所以我们可以设置延迟启动Bean对象

scope(作用域)

Spring容器最初提供了两种bean的scope：singleton和prototype，但是发布2.0版本以后又引入了另外三种：request，session和global session，但这三种类型只能在Web应用中使用。所以我们使用最多的就是最初的两种。注：Spring也提供自定义范围作用域。

singleton

容器默认Scope。表示在IOC容器中只存在一个实例，所有对该对象的应用都共享这个实例。从IOC容器启动，该对象实例初始化后直到IOC容器销毁退出

prototype

容器每次在接到该类型对象的请求时，每次都会生成一个新的对象给请求方。IOC容器只负责对象实例化和属性配置等工作，一旦生成完毕后，IOC容器不再管理该对象的生命周期。

Bean的容器注解配置

ComponentScan，ComponentScans

@ComponentScan跟@ComponentScans的作用就是扫描路径包下是否包含Component注解的Bean对象，若有的情况下，将其注入到IOC容器中。
ComponentScan跟ComponentScans的区别在于，ComponentScan只能扫描一个路径包下的Bean对象，ComponentScans可以扫描多个。

Component

Component注解表示将当前类注入到IOC容器中去。IOC容器在启动的时候会进行范围扫描，若检测到该类上带有Component注解，则会将该类进行注入。

Configuration

@Configuration注解一般标注在类上，它也包含Component注解，所以也可以当作一个Bean注入到IOC容器中去。功能相当于XML配置文件，其中包含各种Bean的配置类。
注：@Configuration注解对类的配置有要求，不可以是final类型，不可以是匿名类。

Import

xml形式中的<import></import>标签。一般情况下，我们根据不同的模块功能或分层的关系，会将配置信息分门别类。在主配置中若想将其他依赖的配置一起加载，则使用Import注解。

Controller，Service，Repository

Controller，Service，Respository三个注解跟Component注解类似，只不过是针对更为具体的场景。Controller表示控制层，Service表示服务层，Respository表示持久层。

Conditional

@Conditional注解是Spring4新提供的注解，作用是按照一定的条件进行判断，选择将Bean注入到容器中去。

Value

@Value通常用于注入外部属性。一般是注入成员属性，或者利用构造器进行注入。
注：无法注入静态成员属性

Bean的生命周期

前面我们已经基本的了解了IOC容器（控制反转与依赖注入）的基本原理，也知道IOC容器其实就是管理Bean的，但是在管理的过程中，Bean是怎么从创建到销毁的过程我们还不得而知。

 Spring Bean的生命周期的过程为：创建Bean对象 -->填充Bean的成员属性（依赖注入）-->初始化Bean（执行初始化方法）-->销毁Bean对象。

Java注解

事实上，Java有一些注解也可以对Java Bean的生命周期进行增强，例如：@PostConstruct（表示执行初始化方法），@PreDesctroy（销毁前执行）。

注：使用@PostConstruct，@PreDesctroy注解需要引入Javax的注解api依赖包

Bean注解配置initMehod和destroyMethod

@Configuration配置类中用@Bean注解也可以执行初始化方法和对象销毁前执行的方法。

Aware接口

实现Aware接口可以在Bean依赖注入的时候将Spring的资源注入到Bean对象中去。从而Bean对象也可以获取到Spring的内部资源。
BeanNameAware，BeanClassLoaderAware，BeanFactoryAware在初始化Bean时执行
剩下的接口是通过后置处理器BeanPostProcessor前初始化完成。

Bean的生命周期总结

Bean的生命周期分为4个部分。其中初始化方法，销毁Bean对象这2部分又分为多个小部分。
注：Bean的生命周期中还涉及到BeanPostProcessor（后置处理器）部分，这部分跟AOP有很大的关系.
