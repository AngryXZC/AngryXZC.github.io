装饰器模式设计模式

发表于 2024-05-17 更新于 2024-05-20 分类于 设计模式 阅读次数： 本文字数： 361 阅读时长 ≈ 1 分钟
装饰器模式是一种结构型设计模式，它允许你通过将对象放入包含行为的特殊封装对象中来为原对象绑定新的行为。

装饰器模式设计模式


classDiagram
class Component{
    <<abstract>>
    +Operation()
}
class ConcreteComponent{
    +Operation()
}
class Decorator{
    <<abstract>>
    -component:Component
    + setComponent(component:Component)
    +Operation()
}
class ConcreteDecoratorA{
    -addedState:string
    +Operation()
}
class ConcreteDecoratorB{
    +Operation()
    -AddedBehavior()
}
Component <|-- ConcreteComponent
Component <|-- Decorator
Decorator <|-- ConcreteDecoratorA
Decorator <|-- ConcreteDecoratorB
Decorator o--> Component:component

note for Component "抽象构件类，定义一个对象接口，可以给这些对象动态地添加职责"
note for Decorator "装饰抽象类，继承了Component,从外类来扩展Component类的功能，但对于Component来说，是无需知道Decorator的存在的"
note for ConcreteDecoratorA "具体装饰类，起到给Component添加职责的功能,和ConcreteDecoratorB类是一样的，只是起到不同的作用"
note for ConcreteComponent "具体构件类，实现了Component接口的具体对象"

装饰模式的总结

装饰模式是一种结构型设计模式，是为已有功能动态地添加更多功能的一种方式。装饰模式是继承关系的一个替代方案。装饰模式可以在不改变一个对象本身的基础上，在程序运行期间给对象动态地添加职责。

优缺点：

  • 优点：把类中的装饰功能从类中搬移去除，这样可以简化原有的类。有效地把类的核心职责和装饰功能区分开了，而且可以去除相关类中重复的装饰逻辑。
  • 缺点：装饰模式会增加许多子类，这样会使得子类变得很多，增加了系统的复杂性。
