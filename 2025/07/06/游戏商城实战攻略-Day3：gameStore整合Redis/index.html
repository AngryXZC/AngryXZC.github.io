Spring Cache 介绍

Spring Cache 是 Spring 提供的缓存抽象，允许开发者在应用中轻松集成缓存机制。它支持多种缓存实现，如 Redis、Ehcache、Caffeine 等。

常用注释

@EnableCaching：启用缓存功能,一般放在启动类上。

@Cacheable：使用该注解的方法当缓存存在时，会从缓存中获取数据而不执行方法，当缓存不存在时，会执行方法并把返回结果存入缓存中。一般使用在查询方法上，可以设置如下属性：

  • value：缓存名称
  • key：缓存的键，可以使用 SpEL 表达式
  • condition：条件表达式，满足条件时才缓存
  • unless：条件表达式，不满足条件时不缓存

@CachePut：使用该注解的方法每次执行时都会把返回结果存入缓存中。一般使用在新增方法上，可以设置如下属性：

  • value：缓存名称
  • key：缓存的键，可以使用 SpEL 表达式
  • condition：条件表达式，满足条件时才缓存
  • unless：条件表达式，不满足条件时不缓存

@CacheEvict：使用该注解的方法执行时会清空指定的缓存。一般使用在更新或删除方法上，可以设置如下属性：

  • value：缓存名称
  • key：缓存的键，可以使用 SpEL 表达式
  • allEntries：是否清空所有缓存项
  • condition：条件表达式，满足条件时才清空缓存

目前项目中没有使用 @EnableCaching。 而是使用RedisTemplate直接操作 Redis 的代码，不依赖 Spring Cache 机制

RedisTemplate 与 Spring Cache 的区别总结

一句话总结

  • RedisTemplate 是底层 API，操作灵活，适用于自定义缓存逻辑。
  • Spring Cache 是更高层的抽象，使用注解简化缓存，适合标准化的读写缓存场景。

一、RedisTemplate 是什么？

RedisTemplate 是 Spring Data Redis 提供的 Redis 客户端操作模板类，可直接操作 Redis 的各种数据结构：

  • String（字符串）
  • Hash（哈希）
  • List（列表）
  • Set（集合）
  • ZSet（有序集合）

优点

  • 灵活强大，支持多种 Redis 数据结构；
  • 可定制序列化、过期时间、事务等；
  • 适合实现分布式锁、排行榜等高级功能。

缺点

  • 需要写大量样板代码；
  • 与业务逻辑耦合较重，不易维护。

二、Spring Cache 是什么？

Spring Cache 是 Spring Framework 提供的一套通用缓存抽象接口，配合注解快速开启缓存功能。

常用注解包括：

  • @Cacheable：方法调用前先查缓存；
  • @CachePut：方法执行后更新缓存；
  • @CacheEvict：方法执行后清除缓存。

优点

  • 使用注解，开发简洁；
  • 易于维护，统一配置缓存策略；
  • 支持命名空间、自动失效等功能。

缺点

  • 粒度较粗，不适合复杂结构；
  • 灵活性不如 RedisTemplate。

三、使用对比总结

特性 RedisTemplate Spring Cache
封装层次 底层 Redis 操作 高层缓存抽象接口
使用方式 编码实现 注解驱动
灵活性 高，支持多种数据结构 中等，主要用于对象缓存
适用场景 排行榜、分布式锁、复杂缓存等 读多写少的查询缓存
配置难度 较高 较低
缓存刷新/过期控制 手动实现 可通过注解参数配置

实战建议

场景 推荐使用
数据结构复杂，如 List/Set/ZSet RedisTemplate
简单的查询缓存 Spring Cache
需要精细控制缓存行为 RedisTemplate
快速为服务添加标准缓存 Spring Cache

组合使用建议

在实际项目中，建议结合使用：

  • 用 Spring Cache 管理常规查询缓存；
  • 用 RedisTemplate 处理高级 Redis 操作，如计数器、排行榜、分布式锁等。

Jedis 和Lettuce

Jedis 和 Lettuce 都是 Java 中常用的 Redis 客户端库，但它们有一些关键区别：

特性 Jedis Lettuce
连接方式 基于阻塞 I/O，单线程模型 基于非阻塞 I/O，支持多线程和异步操作
连接池 内置连接池，使用简单 支持多种连接池实现，如 Commons Pool、Lettuce 自带连接池等
异步支持 不支持异步操作 支持异步和反应式编程
性能 在高并发场景下性能较差 在高并发场景下性能更好，支持多线程并发操作
依赖 依赖较少，轻量级 依赖较多，功能更强大
使用场景 简单的 Redis 操作，适合小型项目 高性能、高并发的 Redis 操作，适合大型项目
